---

## 1. Цель Gold Standard

Gold Standard (золотой стандарт) — это набор из 50 вручную проверенных и отредактированных примеров, которые служат:

- **Эталоном качества** для оценки автоматической разметки
- **Базой для синтетической генерации** (на их основе LLM генерирует дополнительные данные)
- **Тестовым набором** для оценки качества дообученной модели (человеческая оценка)

> ⚠️ Качество Gold Standard напрямую определяет качество всего датасета. Если в GS есть ошибки, они умножатся при синтетической генерации.

## 4. Формат записи

Каждый пример — одна строка в JSONL-файле `dataset/data/gold_standard.jsonl`.

### Структура:

```json
{
  "messages": [
    {
      "role": "system",
      "content": "Ты — интеллектуальный тьютор для студентов среднего профессионального образования по специальности 15.02.14..."
    },
    {
      "role": "user",
      "content": "<запрос студента или задание для ИИ>"
    },
    {
      "role": "assistant",
      "content": "<идеальный ответ, который должен давать ИИ-тьютор>"
    }
  ],
  "metadata": {
    "task_type": "summary|quiz|chat|explanation",
    "subject": "industrial_auto|measurement_systems",
    "difficulty": "easy|medium|hard",
    "topic": "Тема лекции (строка)",
    "source": "manual",
    "source_file": "Имя исходного файла лекции",
    "annotator": "Фамилия И.О.",
    "reviewed_by": "Фамилия И.О. (после проверки)",
    "review_date": "YYYY-MM-DD"
  }
}
```

### Отличия от обычных записей:

1. **`source`: всегда `"manual"`** — пометка, что это ручная разметка
2. **`annotator`**: ФИО разметчика
3. **`reviewed_by`**: заполняется валидатором после проверки (пусто до проверки)
4. **`review_date`**: дата проверки

## 5. Правила разметки по типам задач

### 5.1 Summary (конспект лекции)

**Вход (user):** Текст лекции (или ссылка на файл)

**Требования к ответу (assistant):**
- Структурированный конспект с заголовками и подзаголовками
- 500–1500 слов (зависит от объёма лекции)
- Ключевые термины **выделены жирным**
- Каждый раздел содержит:
  - Определение/суть (1-2 предложения)
  - Ключевые характеристики (список)
  - Практическое применение (1-2 предложения)
- НЕТ галлюцинаций — только факты из исходной лекции
- НЕТ воды — каждая строка несёт смысл

**Шаблон запроса (user):**
```
Составь структурированный конспект лекции по теме «[ТЕМА]».
Исходный материал:

[ТЕКСТ ЛЕКЦИИ]
```

**Шаблон ответа (assistant):**
```markdown
# [ТЕМА]

## 1. [Раздел 1]

**[Термин]** — [определение].

Характеристики:
- [характеристика 1]
- [характеристика 2]
- [характеристика 3]

Практическое применение: [описание].

## 2. [Раздел 2]
...
```

### 5.2 Quiz (генерация теста)

**Вход (user):** Тема + количество вопросов + сложность

**Требования к ответу (assistant):**
- Формат: вопрос → 4 варианта ответа → правильный ответ → пояснение
- Каждый вопрос проверяет понимание, не механическое запоминание
- Варианты ответов правдоподобны (дистракторы похожи на правильный)
- Пояснение объясняет, почему ответ правильный и почему остальные неверны

**Шаблон запроса (user):**
```
Создай тест по теме «[ТЕМА]» для студентов СПО.
Количество вопросов: [N]
Сложность: [easy|medium|hard]
```

**Шаблон ответа (assistant):**
```markdown
## Тест: [ТЕМА]

### Вопрос 1
[Формулировка вопроса]

а) [Вариант А]
б) [Вариант Б]
в) [Вариант В]
г) [Вариант Г]

**Правильный ответ:** [буква]

**Пояснение:** [объяснение]
```

### 5.3 Chat (диалог с тьютором)

**Вход (user):** Вопрос студента по теме

**Требования к ответу (assistant):**
- Язык: доступный для студентов СПО (без излишней академичности)
- Объём: 100–300 слов
- Структура: ответ → пример → связь с практикой
- Если вопрос слишком широкий — разбить на подтемы
- НЕТ копипаста из учебника — пересказ своими словами

**Шаблон запроса (user):**
```
Объясни, как работает [ПОНЯТИЕ] в контексте [КОНТЕКСТ].
```

### 5.4 Explanation (объяснение понятия)

**Вход (user):** Термин или понятие для объяснения

**Требования к ответу (assistant):**
- Аналогия из реальной жизни (помогает понять суть)
- Строгий вариант объяснения (1-2 предложения)
- Формула/схема (если применимо)
- Типичные ошибки студентов при изучении данного понятия

## 6. Чек-лист качества для каждого примера

Перед сохранением каждого примера проверьте:

### Формат
- [ ] Валидный JSON (проверить через `python -m json.tool`)
- [ ] Наличие всех обязательных полей (messages, metadata)
- [ ] Порядок ролей: system → user → assistant
- [ ] Поле `source` = `"manual"`
- [ ] Поле `annotator` заполнено

### Контент
- [ ] Ответ ассистента соответствует запрошенному типу задачи
- [ ] Фактически корректно (нет выдуманных данных, формул, дат)
- [ ] Термины использованы корректно
- [ ] Язык понятен для студента СПО (2-й курс)
- [ ] Нет повторов одного и того же содержания
- [ ] Длина ответа в допустимом диапазоне (см. п.5)

### Специфичное для summary
- [ ] Все ключевые темы лекции покрыты
- [ ] Структура логичная и последовательная
- [ ] Термины выделены

### Специфичное для quiz
- [ ] Все вопросы уникальны (нет дубликатов)
- [ ] Дистракторы правдоподобны
- [ ] Правильные ответы корректны
- [ ] Пояснения информативны

## 7. Процесс валидации (2-й уровень)

После создания всех 50 примеров, валидатор (Бардаков Д.Н.) проводит проверку:

### Шаг 1: Автоматическая валидация
```bash
cd /home/z/my-project/al
python scripts/validate_annotations.py --data-dir dataset/data/ --strict
```

Ожидаемый результат: 0 ошибок, минимум предупреждений.

### Шаг 2: Ручная проверка каждого примера

Для каждого из 50 примеров:
1. Открыть в JSONL-редакторе
2. Сравнить ответ ассистента с исходной лекцией (для summary)
3. Проверить корректность фактов (для quiz)
4. Оценить понятность (для chat/explanation)
5. Заполнить `reviewed_by` и `review_date`

### Шаг 3: Оценка согласованности (Inter-Annotator Agreement)

Если два разметчика работали параллельно, проверить:
- Выбор 5 случайных примеров из набора каждого разметчика
- Независимая оценка одним валидатором
- Расчёт Cohen's Kappa (целевой ≥ 0.8)
