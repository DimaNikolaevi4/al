## 3. Анализ «обрыва скачивания»

Гипотеза: модель не разворачивается из-за обрыва при долгом скачивании GGUF.

### Почему wget может обрываться на Qubu

- **14.3 ГБ** — огромный файл для однопоточного HTTP-скачивания
- **Таймаут TCP-соединения** — HuggingFace может сбрасывать долгие подключения
- **Ограничение диска** — если диск на Qubu < 20 ГБ, wget упадёт с `No space left on device`
- **Прокси/балансировщик Qubu** — может обрывать долгие загрузки
- **wget без retry** — в startup_script.sh стоит `wget -c` (continue), но при первом запуске нет кэша
- **Таймаут wget по умолчанию** — 900 сек (15 мин) неактивности; при throttled скорости wget может «зависнуть» и быть убит по таймауту билда

### Вывод: обрыв wget — наиболее вероятная причина провала деплоев

## 4. Расчёт времени билда (по компонентам)

Что происходит при деплое (последовательно):

| Этап | Время (оценка) | Примечание |
|------|----------------|------------|
| **BentoML packaging** (bentoml build) | 2-5 мин | Зависимости, Docker image |
| **pip install bentoml** | 1-2 мин | ~50 MB |
| **pip install llama-cpp-python** (pre-built wheel) | 1-2 мин | ~200 MB wheel, без компиляции |
| **wget GGUF модели** | **10-25 мин** | Зависит от скорости канала Qubu |
| **Загрузка модели в RAM + VRAM** | 2-5 мин | mmap + GPU offload |
| **BentoML serve — холодный старт** | 1-3 мин | Инициализация сервиса |
| **ИТОГО** | **17-42 мин** | В лучшем случае впритык |

## 5. Бюджет времени билда (50 минут = 3000 секунд)

### Стратегия А: Скачать на сервере (wget) — ТЕКУЩАЯ

```
BentoML build + pip install     : ~6-8 мин
pip install wheel (200 MB)      : ~2 мин
wget 14.3 GB из HuggingFace    : ~15-25 мин  <-- ПРОБЛЕМНО
Загрузка модели в память        : ~3-5 мин
bentoml serve старт             : ~2 мин
-------------------------------------------
ИТОГО                           : ~28-42 мин  <-- ВПРИТЫК, может упасть
```

### Стратегия Б: Git LFS (модель уже в репо Qubu)

```
BentoML build + pip install     : ~6-8 мин
pip install wheel (200 MB)      : ~2 мин
git pull LFS (модель из репо)   : ~10-20 мин  <-- Быстрее чем HF?
Загрузка модели в память        : ~3-5 мин
bentoml serve старт             : ~2 мин
-------------------------------------------
ИТОГО                           : ~23-37 мин  <-- ЛУЧШЕ, но всё ещё рискованно
```

### Стратегия В: Модель 4-5 ГБ через wget

```
BentoML build + pip install     : ~6-8 мин
pip install wheel (200 MB)      : ~2 мин
wget 5 GB из HuggingFace       : ~5-10 мин   <-- НАДЁЖНО
Загрузка модели в память        : ~1-2 мин
bentoml serve старт             : ~1 мин
-------------------------------------------
ИТОГО                           : ~15-23 мин  <-- КОМФОРТНО
```

### Критические исправления перед любым деплоем

1. Починить код на Qubu — текущий код СЛОМАН (конкатенация двух версий)
2. Починить requirements — сейчас мусорная строка
3. Починить startup_script — обрезан посередине
4. Использовать `wget -c --timeout=30 --tries=3` — retry при обрыве
5. Добавить `curl -fSL -o` как fallback после wget
6. Слить GGUF в Git LFS — чтобы не зависеть от HuggingFace при каждом деплое
7. Уменьшить n_ctx с 2048 до 1024 для экономии VRAM (если нужно)

## 11. Ограничения BentoML (дополнительная информация)

- **Нет жёсткого лимита на размер модели** в самом BentoML
- **Container image**: типичный LLM контейнер ~6.7 ГБ compressed, до 20+ ГБ с моделью
- **Docker layer limit**: большинство registry ограничивают слои до 5-10 ГБ
- **llama-cpp-python**: загружает ВЕСЬ GGUF в память через mmap; RAM ~ размер файла + 1-2 ГБ
- **Best practice**: НЕ запекать модель в Docker image; скачивать при старте
- **traffic.timeout** в `@bentoml.service`: по умолчанию 60 сек, для LLM рекомендуют 300-600 сек
