#### Результат 1.1.4 — Формат датасета

Определено на основе: tutor.py (метод `chat()` использует `apply_chat_template()`), библиотеки Transformers/Hugging Face (стандарт SFTTrainer), чат-шаблона Mistral Small 3.1.

**Формат файла:** JSONL (один JSON-объект на строку, кодировка UTF-8 без BOM)

**Структура каждой строки:**

```json
{
  "messages": [
    {"role": "system", "content": "<системный промпт для типа задачи>"},
    {"role": "user", "content": "<входные данные (текст лекции, вопрос)>"},
    {"role": "assistant", "content": "<ожидаемый ответ модели>"}
  ],
  "metadata": {
    "task_type": "<тип задачи: summary_basic, quiz_multiple_choice и т.д.>",
    "subject": "<предмет: industrial_auto, measurement_systems и т.д.>",
    "difficulty": "<сложность: basic, intermediate, advanced>",
    "topic": "<тема занятия>",
    "source": "<источник: lecture, textbook, moodle>"
  }
}
```

**Обязательные поля:** `messages` (массив из 3 элементов: system → user → assistant)

**Опциональные поля:** `metadata` (для аналитики и балансировки, не используется при обучении)

**Соответствие методам tutor.py:**

| Поле messages | Метод tutor.py | Роль |
|---|---|---|
| `system` | `_build_summary_prompt()` / промпт в `generate_quiz()` | Инструкция модели: что делать |
| `user` | `lecture_text` аргумент методов | Текст лекции / вопрос студента |
| `assistant` | Return value методов | Ожидаемый результат генерации |

**Пример строки для T1 (конспект базовый):**

```json
{
  "messages": [
    {"role": "system", "content": "Ты — ИИ-тьютор для студентов техникума. Сделай краткий структурированный конспект лекции. Выдели основные тезисы, используй маркированные списки."},
    {"role": "user", "content": "Тема: Основы промышленной автоматизации. Промышленная автоматизация — это применение систем управления и информационных технологий для управления промышленными процессами. Основные компоненты: ПЛК, датчики, исполнительные механизмы, SCADA-системы."},
    {"role": "assistant", "content": "## Конспект: Основы промышленной автоматизации\n\n**Промышленная автоматизация** — управление технологическими процессами с помощью систем управления и ИТ.\n\n### Основные компоненты:\n- **ПЛК** (программируемый логический контроллер) — «мозг» автоматики, выполняет программу управления\n- **Датчики** — измеряют параметры процесса (температура, давление, расход)\n- **Исполнительные механизмы** — выполняют команды контроллера (клапаны, моторы)\n- **SCADA-система** — интерфейс оператора для мониторинга и управления"}
  ],
  "metadata": {
    "task_type": "summary_basic",
    "subject": "industrial_auto",
    "difficulty": "basic",
    "topic": "Основы промышленной автоматизации",
    "source": "lecture"
  }
}
```

**Примечание по 1.1.5 (🔶):** Формат `messages` с ролями `system/user/assistant` совместим с `tokenizer.apply_chat_template()` из tutor.py, что решает задачу 1.1.5. Для применения достаточно: `tokenizer.apply_chat_template(row["messages"], tokenize=False)`.

**1. Методика оценки качества источников**

Каждый источник (учебник, методическое пособие, конспект лекции, тест) оценивается по 6 критериям по шкале 1–5 баллов:

| # | Критерий | Описание | Вес |
|---|----------|----------|:---:|
| K1 | Актуальность | Год издания ≤5 лет (5 баллов), ≤10 лет (4 балла), >10 лет (2 балла). Обновлённые издания — +1 балл. | 20% |
| K2 | Соответствие ФГОС СПО | Наличие в УМК дисциплины (ФГР/ФОП). Покрытие mandatory-компетенций из ФГОС 15.02.14. | 25% |
| K3 | Полнота покрытия темы | Соответствие рабочей программе дисциплины (РПД). Наличие теории, примеров, задач, иллюстраций. | 20% |
| K4 | Доступность изложения | Уровень языка соответствует СПО (без излишнего академизма). Наличие примеров, определений, схем. | 15% |
| K5 | Структурированность | Наличие оглавления, глоссария, контрольных вопросов. Логическая последовательность изложения. | 10% |
| K6 | Цифровая доступность | Наличие электронного формата (PDF/Word). Пригодность для оцифровки (OCR-качество сканов ≥95%). | 10% |

**Формула итоговой оценки:** O = K1×0.20 + K2×0.25 + K3×0.20 + K4×0.15 + K5×0.10 + K6×0.10

**Порог включения в датасет:** O ≥ 3.5 баллов (источники ниже порога — только как вспомогательные, не для основного датасета).

**2. Критерии качества тестов для включения в датасет**

Каждый тест/вопрос из Moodle оценивается по 5 критериям (шкала 1–5):

| # | Критерий | Описание | Вес |
|---|----------|----------|:---:|
| C1 | Техническая корректность | Валидность разметки в Moodle (правильный ответ указан, нет конфликтов вариантов, все поля заполнены). | 20% |
| C2 | Содержательная точность | Фактологическая и терминологическая правильность. Соответствие актуальным стандартам (ГОСТ, ПУЭ, учебники из Результата 2.1.2). | 30% |
| C3 | Соответствие уровню СПО | Сложность соответствует профилю студента СИТ (Результат 1.1.3: 2–3 курс, базовые знания физики и информатики). Не слишком простой, не требует высшей математики. | 20% |
| C4 | Наличие обратной связи | Объяснение правильного/неправильного ответа (Moodle: поле «Общая обратная связь»). Ценно для датасета — модель учится не только отвечать, но и объяснять. | 15% |
| C5 | Тематическая привязка | Вопрос привязан к конкретной теме и дисциплине (S1–S4). Не «общий вопрос без контекста». | 15% |

**Формула:** Q = C1×0.20 + C2×0.30 + C3×0.20 + C4×0.15 + C5×0.15

**Порог включения:** Q ≥ 3.5 баллов. Вопросы с Q < 3.5 — отправляются на доработку преподавателю или исключаются.

**3. Детектор типа PDF (текстовый vs. скан)**

Логика маршрутизации:
- Извлечь текст со всех страниц через PyMuPDF (`page.get_text()`)
- Если >70% страниц содержат >50 символов текста — считать PDF текстовым → использовать pdfplumber/PyMuPDF
- Иначе — считать сканом → использовать Tesseract OCR (или EasyOCR)

**4. Правила очистки текста**

| Этап | Правило | Пример |
|---|---|---|
| Мусорные символы | Удалить zero-width space, BOM, non-breaking space (заменить на пробел), form-feed | Удаляются через regex |
| Нумерация страниц | Удалить числа на отдельных строках (1-50) в начале/конце страницы | "47\nТекст" → "Текст" |
| Артефакты OCR | Заменить множественные пробелы (>2) на один, лишние переносы (>2) на 2 | Автоматическая |
| Колонтитулы | Удалить строки, повторяющиеся на >50% страниц | Автоматическая |
| Гиперссылки | Сохранить текст ссылки, удалить URL | "[текст](url)" → "текст" |

| Элемент исходного документа | Markdown-эквивалент | Правило |

**Объём ручной проверки:** 20% от общего числа лекций (каждая пятая). При 50 лекциях по S1 + 30 по S2 = 16 лекций на ручную проверку.

---

**6. Порядок применения**

```
Оцифровка (2.2.3) → Автоматическая проверка → Ручная проверка (20% выборки) → Сохранение в raw/
```

Автоматическая проверка является обязательным этапом для всех файлов. Ручная проверка применяется к выборке (20%) и к файлам, где автоматическая проверка нашла >10 issues.

**4.2.6 — Верификация JSONL формата**

```python
import json

def validate_jsonl(filepath):
    errors = []
    with open(filepath, 'r', encoding='utf-8') as f:
        for i, line in enumerate(f, 1):
            line = line.strip()
            if not line:
                continue
            try:
                obj = json.loads(line)
                if 'messages' not in obj:
                    errors.append(f'Line {i}: missing "messages" field')
                elif len(obj['messages']) != 3:
                    errors.append(f'Line {i}: expected 3 messages, got {len(obj["messages"])}')
                elif not all(m.get('role') in ('system','user','assistant') for m in obj['messages']):
                    errors.append(f'Line {i}: invalid role in messages')
            except json.JSONDecodeError as e:
                errors.append(f'Line {i}: JSON error: {e}')
    return errors
```

#### Результат 1.1.5 — Формат чат-шаблона Mistral Small 3.1

Определено на основе: документации Mistral AI (docs.mistral.ai), спецификации Hugging Face (`mistralai/Mistral-Small-24B-Instruct-2501`), метода `chat()` в tutor.py (использует `tokenizer.apply_chat_template()`).

---

**1. Модель и чат-шаблон**

| Параметр | Значение |
|---|---|
| Модель | `mistralai/Mistral-Small-24B-Instruct-2501` |
| Архитектура | Mistral Small (24B params), Instruct-вариант |
| Контекстное окно | ~32 000 токенов |
| Поддерживаемые роли | `system`, `user`, `assistant` |
| Специальные токены | `<s>`, `</s>`, `[INST]`, `[/INST]` |

---

**2. Формат чат-шаблона**

Mistral Small 3.1 (Instruct) использует формат на основе токенов `[INST]` / `[/INST]`:

```
<s>[INST] {System Prompt} [/INST]
{Assistant Message 1}</s>
<s>[INST] {User Message 1} [/INST]
{Assistant Message 2}</s><s>[INST] {User Message 2} [/INST]
...
```

**Ключевые правила:**
- Каждое сообщение оборачивается в `[INST] ... [/INST]`
- System prompt помещается в первое сообщение (без ответа модели перед ним)
- Ответы модели размещаются между `[/INST]` и `<s>` (или `</s>`)
- Для многоходовых диалогов каждое сообщение чередуется: user `[INST]...[/INST]` → assistant

**Пример для 3 сообщений (system + user + assistant):**

```
<s>[INST] Ты — ИИ-тьютор для студентов техникума. [/INST]
Сделай краткий конспект.</s><s>[INST] Тема: Основы промышленной автоматизации. [/INST]
## Конспект: Основы промышленной автоматизации

### Основные компоненты:
- ПЛК — программируемый логический контроллер
- Датчики — измеряют параметры процесса
</s>
```

---

**3. Роли и их поведение**

| Роль | Описание | Поведение в шаблоне |
|---|---|---|
| `system` | Системная инструкция для модели (роль, поведение, стиль) | Помещается в первый `[INST]...[/INST]` без предшествующего ответа |
| `user` | Входные данные: текст лекции, вопрос студента | Оборачивается в `[INST]...[/INST]` |
| `assistant` | Ожидаемый ответ модели (для дообучения) | Размещается после `[/INST]` и до следующего `<s>` |

---

**4. Использование в tutor.py и датасете**

Метод `chat()` в tutor.py использует `tokenizer.apply_chat_template()`:

```python
messages = [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
]
prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
```

`apply_chat_template()` автоматически преобразует массив `messages` в формат Mistral (с `[INST]` / `[/INST]`). Для датасета это означает:

| Формат датасета | Конвертация в шаблон Mistral |
|---|---|
| `messages[0].role = "system"` | `<s>[INST] {content} [/INST]` |
| `messages[1].role = "user"` | (продолжение) `{content} [/INST]` |
| `messages[2].role = "assistant"` | (продолжение) `{content}</s>` |

**Команда для конвертации в обучающий формат:**

```python
from transformers import AutoTokenizer
tokenizer = AutoTokenizer.from_pretrained("mistralai/Mistral-Small-24B-Instruct-2501")
text = tokenizer.apply_chat_template(row["messages"], tokenize=False)
```

---

**5. Специальные случаи**

| Ситуация | Решение |
|---|---|
| Многоходовой диалог (T7: chat_explanation) | Добавить несколько пар `user`/`assistant` в `messages`. Шаблон автоматически расширяется. |
| Без system prompt (не рекомендуется) | Пропустить элемент с `role="system"`. Первый `[INST]` будет содержать только user-сообщение. |
| Длинный контекст (>32K токенов) | `apply_chat_template()` не обрезает — обрезка выполняется через `tokenizer(..., truncation=True, max_length=32768)` |

---

**6. Вывод**

Формат чат-шаблона Mistral Small 3.1 полностью совместим с форматом датасета из Результата 1.1.4 (массив `messages` с ролями `system`/`user`/`assistant`). Конвертация выполняется автоматически через `apply_chat_template()`. Не требуется ручное форматирование токенов `[INST]`/`[/INST]` — библиотека Transformers обрабатывает это.

```python
def check_topic_coverage(dataset_path, topics_list):
    dataset_topics = set()
    for line in open(dataset_path):
        obj = json.loads(line)
        dataset_topics.add(obj['metadata']['topic'])
    all_topics = set(topics_list)
    covered = dataset_topics & all_topics
    missing = all_topics - dataset_topics
    return {"coverage_pct": len(covered)/len(all_topics)*100, "covered": list(covered), "missing": list(missing)}
```

**1. Скрипт `dataset/validate.py` — Автоматическая валидация (8.1.1–8.1.6)**

Скрипт выполняет комплексную проверку трёх сплитов (train/val/test) и возвращает exit-код 0 (пройдено) или 1 (ошибки).

```bash
# Полная валидация с сохранением отчёта
python dataset/validate.py \
    --train dataset/data/train.jsonl \
    --val dataset/data/val.jsonl \
    --test dataset/data/test.jsonl \
    --output dataset/data/stats.json
```

| Задача | Что проверяется | Метод | Критерий прохождения |
|:---:|---|---|---|
| 8.1.1 | JSONL-формат каждой строки | `json.loads()` + проверка структуры messages | 0 ошибок парсинга; наличие system/user/assistant |
| 8.1.2 | Длина токенов (оценка: слова × 1.3) | Проверка user, assistant, system по лимитам | system ≤500; user: 128–2048; assistant: 64–4096; total ≤32K |
| 8.1.3 | Кросс-дубликаты между сплитами | SHA-256 хеш поля `assistant` | 0 пересечений между train, val, test |
| 8.1.4 | Баланс типов задач в каждом сплите | Counter по `metadata.task_type` | Каждый тип: ≥5% от сплита |
| 8.1.5 | Статистический анализ | Среднее/мин/макс/медиана токенов; уникальные темы | Отчёт сохраняется в stats.json |
| 8.1.6 | Итоговая валидация | Общий exit-код скрипта | Все проверки 8.1.1–8.1.5 пройдены → exit 0 |

**Формат отчёта валидации** (`stats.json`):
```json
{
  "model": "Mistral Small 3.1 (24B)",
  "context_window": 32768,
  "status": "PASSED",
  "train_jsonl_errors": 0,
  "train_token_warnings": 0,
  "cross_duplicates": 0,
  "statistics": {
    "train": {"records": 400, "total_tokens": {"avg": 892, "min": 256, "max": 4096}},
    "val": {"records": 50, "total_tokens": {"avg": 850, "min": 300, "max": 3800}},
    "test": {"records": 50, "total_tokens": {"avg": 870, "min": 280, "max": 3950}}
  }
}
```

**2. Методика ручной проверки (8.2.1–8.2.3)**

Каждый пример из выборки оценивается по 6 критериям (из Результата 3.2.2):

| # | Критерий | Что проверяется | Шкала | Порог |
|---|---|---|:---:|:---:|
| V1 | Точность терминов | Соответствие глоссарию (Результат 4.2.3) | 1–5 | ≥4 |
| V2 | Корректность фактов | Числа, названия, формулы | 1–5 | ≥4 |
| V3 | Стиль изложения | Уровень СПО (Результат 1.1.3): без канцелярита | 1–5 | ≥4 |
| V4 | Полнота ответа | Покрытие всех аспектов запроса | 1–5 | ≥4 |
| V5 | Формат | Markdown-разметка, структура | 1–5 | ≥4 |
| V6 | Длина | В пределах лимитов токенов | 1/0 | 1 |

| Сплит | Объём выборки | Ответственный | Действие при обнаружении ошибок |
|---|:---:|---|---|
| train (8.2.1) | минимум 20 примеров (случайная выборка) | Мышанская Н.Г. | Исправить или удалить из датасета |
| val (8.2.2) | минимум 10 примеров | Мышанская Н.Г. | Исправить или заменить |
| test (8.2.3) | минимум 10 примеров | Мышанская Н.Г. | Исправить или заменить (test не трогается при обучении) |

**Форма проверки:**
Для каждого примера эксперт заполняет таблицу с оценками V1–V6. Примеры с V ≤ 3 по любому критерию возвращаются на доработку. Если доля брака в выборке >20% — вся выборка отправляется на ревью.
